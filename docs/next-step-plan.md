# 文档对齐后的最小启动顺序（Next Step Plan）

## 不先做什么
- 不先做 UI 页面扩展
- 不先做 FLARE Core 流程改造
- 不先做全品类扩展
- 不先做复杂自动化执行链路

## 先做什么
1. 锁定 P0 两个场景（活动、礼品）的字段与补问策略
2. 锁定供应商评估框架与 RFX 路由规则
3. 锁定 P0 产物输出结构
4. 以文档为输入再进入最小实现

## 第一批建议落地文件（开发输入）
1. `docs/activity-procurement-pack.md`
2. `docs/gift-customization-pack.md`
3. `docs/supplier-evaluation-framework.md`
4. `docs/rfx-routing-spec.md`
5. `docs/artifact-output-spec.md`

## 第一批开发顺序建议
1. **字段与补问实现**（先保障输入可靠）
2. **分析模板实现**（形成中间可解释结果）
3. **寻源与评分实现**（形成可比较候选）
4. **RFX路由实现**（形成动作建议）
5. **产物构建实现**（形成业务交付）

## 为什么这个顺序最稳
- 先把输入做准，再做推理与推荐，最后再做输出包装
- 避免先做 UI 导致“展示先行、规则滞后”
- 每一步都有可验证中间产物，风险可控

## 输入 / 输出 / 依赖 / 暂不做

### 输入
- 已完成的实例边界与领域资产文档

### 输出
- 可执行的最小开发启动顺序

### 依赖
- P0 场景文档与规则文档已冻结

### 暂不做
- 开发阶段一次性并行大规模改造
