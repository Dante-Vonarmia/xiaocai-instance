# 采购需求梳理输出架构

> 这是小采行业包的需求梳理输出架构。模型负责理解用户输入、历史参考和上传模板；Kernel/Canvas 负责把模型结果归一化为以下可编辑文档结构。不要输出旧的状态进度块，也不要把缺失字段做成逐项问答。

## 输出质量要求

- 先给业务可用的正文，再把不确定项后置；不要只列字段缺口。
- 每个判断都区分“已确认、当前假设 / 候选、待验证”。
- 能从上下文合理推导的内容可写成候选，但不得伪装为已确认事实。
- 采购目的、使用场景、产品/服务、数量、预算、交付地点和交付时间必须放在同一个项目概述口径里，不要割裂成独立的采购任务树。
- 只有用户明确提出多品类、多采购包或多供应商路线时，才在项目概述或规格与约束中补充二级说明；默认不单独拆出采购任务、规格画像或场景偏好等独立结构。
- 文档应能直接进入后续需求分析、RFX 策略、询价/评审准备。
- 结构化视图需要能映射到项目概述、规格与约束、需求部门信息和决策口径，不要默认生成过重层级。

## 【项目名称】

帮用户取一个清晰、可修改的项目名称。没有明确名称时，基于采购目的和使用场景生成候选名称，并标注“候选”。

## 一页式结论

用 3-5 条先说明当前最重要的采购判断：

- 当前采购目标与业务场景是什么。
- 当前采购对象 / 产品服务是什么，数量、预算和交付口径是否已清楚。
- 哪些信息已经足够支撑下一步分析或 RFX 准备。
- 哪些缺口会直接影响预算、供应商选择、交付、验收或合同风险。
- 当前建议路径：继续补齐需求、进入需求分析、准备 RFI/RFP/RFQ，或先做供应市场验证。

## 一、项目概述

用一个统一基本信息块承载本次需求，不要把同一批信息拆到采购任务或其他独立结构中。

| 字段 | 当前口径 | 状态 |
| --- | --- | --- |
| 项目名称 | 待补充 / 候选 | 待确认 |
| 采购目的 | 待补充 / 候选 | 待确认 |
| 使用场景 | 待补充 / 候选 | 待确认 |
| 产品/服务 | 待补充 / 候选 | 待确认 |
| 数量和单位 | 待补充 / 候选 | 待确认 |
| 预算金额 / 币种 | 待补充 / 候选 | 待确认 |
| 交付地点 | 待补充 / 候选 | 待确认 |
| 交付时间 | 待补充 / 候选 | 待确认 |

## 二、规格与约束

只保留会影响报价、交付、质量、验收或合同风险的规格和约束；未知项标注为待补充或候选。

| 维度 | 当前口径 | 待确认点 |
| --- | --- | --- |
| 一级品类 | 待补充 / 候选 | 待确认 |
| 二级品类 | 待补充 / 候选 | 待确认 |
| 规格档位 / 材质 / 颜色 / 环保 / 质保要求 | 待补充 / 候选 | 待确认 |
| 产品/服务说明 | 待补充 / 候选 | 待确认 |
| 特殊要求 | 待补充 / 候选 | 待确认 |
| 验收口径 | 待补充 / 候选 | 待确认 |
| 供应商资质 / 履约要求 | 待补充 / 候选 | 待确认 |

## 三、需求部门信息（如有）

上一轮输入没有时保留待补充，不阻塞正文输出：

- 【需求来源】
- 【需求提出人】
- 【PR 编号】
- 【影响范围（部门/区域）】
- 【关键干系人 / 审批人】

## 四、决策口径与下一步

给出可执行的决策口径，而不是泛泛建议：

- 是否建议进入需求分析；说明依据。
- 是否建议进入 RFI / RFP / RFQ / RFX；说明适用条件。
- 当前已经确认的信息。
- 当前假设 / 候选信息。
- 推荐先确认的 1-3 个关键问题。
- 下一步需要准备的资料、模板或内部确认动作。
