# 采购需求梳理输出架构

> 这是小采行业包的需求梳理输出架构。模型负责理解用户输入、历史参考和上传模板；Kernel/Canvas 负责把模型结果归一化为以下可编辑文档结构。不要输出旧的状态进度块，也不要把缺失字段做成逐项问答。

## 输出质量要求

- 先给业务可用的正文，再把不确定项后置；不要只列字段缺口。
- 每个判断都区分“已确认、当前假设 / 候选、待验证”。
- 能从上下文合理推导的内容可写成候选，但不得伪装为已确认事实。
- 多采购方向必须形成采购任务策略，避免把活动、礼品、传播、住宿交通等混成一个清单。
- 文档应能直接进入后续需求分析、RFX 策略、询价/评审准备。
- 结构化视图需要能映射到项目、采购任务、规格画像、来源、缺口和下一步节点。

## 【项目名称】

帮用户取一个清晰、可修改的项目名称。没有明确名称时，基于采购目的和使用场景生成候选名称，并标注“候选”。

## 一页式结论

用 3-5 条先说明当前最重要的采购判断：

- 当前采购目标与业务场景是什么。
- 建议按哪些采购任务 / 采购方向推进。
- 哪些信息已经足够支撑下一步分析或 RFX 准备。
- 哪些缺口会直接影响预算、供应商选择、交付、验收或合同风险。
- 当前建议路径：继续补齐需求、进入需求分析、准备 RFI/RFP/RFQ，或先做供应市场验证。

## 一、项目概述

必须尽量覆盖：

- 【采购目的】本次采购要解决的问题、达成的业务目标或活动目标。
- 【使用场景】采购结果用于什么场景、对象、活动或业务环节。
- 【预算金额】已知则填写；未知则标注待补充或给出候选预算口径。
- 【交付地点】城市 / 区域 / 全国；未知则标注待补充或候选范围。
- 【交付时间】具体日期、时间窗口或里程碑；未知则标注待补充。
- 【成功标准】本次采购什么情况下算成功，例如效果、质量、按期、成本、合规或满意度。

## 需求部门信息（如有）

上一轮输入没有时保留待补充，不阻塞正文输出：

- 【需求来源】
- 【需求提出人】
- 【影响范围（部门/区域）】
- 【关键干系人 / 审批人】

## 二、细化需求

先给出品类判断：

- 【一级品类】
- 【二级品类】
- 【品类判断依据】说明为什么归入该品类；不确定时列候选品类和原因。

然后按采购对象、服务范围或采购任务输出细化表：

| 产品/服务 | 数量和单位 | 产品/服务说明 | 特殊要求 | 信息状态 |
| --- | --- | --- | --- | --- |
| 待补充 / 候选 | 待补充 / 候选 | 待补充 / 候选 | 待补充 / 候选 | 已确认 / 候选 / 待验证 |

## 规格画像与字段包

不同规格采购、不同服务类型或不同行业场景，要尽量映射到用户真正需要的结构，而不是固定一张通用表。

| 规格维度 | 当前口径 | 来源 | 状态 |
| --- | --- | --- | --- |
| 功能 / 服务范围 | 待补充 / 候选 | 用户输入 / 会话参考 / 模板 / 检索 | 已确认 / 候选 / 待验证 |
| 性能 / 容量 / 档位 | 待补充 / 候选 | 用户输入 / 会话参考 / 模板 / 检索 | 已确认 / 候选 / 待验证 |
| 交付 / 实施 / 执行方式 | 待补充 / 候选 | 用户输入 / 会话参考 / 模板 / 检索 | 已确认 / 候选 / 待验证 |
| 验收 / 服务 / 售后 | 待补充 / 候选 | 用户输入 / 会话参考 / 模板 / 检索 | 已确认 / 候选 / 待验证 |
| 合规 / 安全 / 资质 | 待补充 / 候选 | 用户输入 / 会话参考 / 模板 / 检索 | 已确认 / 候选 / 待验证 |

## 采购任务 / 采购方向（如适用）

如果一个需求同时包含活动执行、礼品、茶歇、传播、住宿交通等不同采购方向，必须拆成多个采购任务或采购方向。

每个采购任务至少保留：

- 采购任务名称 / 方向
- 目标与范围
- 不包含事项 / 边界
- 品类候选与供应商类型
- 预算或数量候选
- 关键交付物
- 验收与交付标准
- 主要风险与依赖
- 待确认点

## 行业 / 场景偏好结构

当用户上传公司模板、历史会话或行业记录里出现偏好的结构时，优先贴合该结构生成正文；不能确认时只作为候选结构。

- 【适用行业 / 场景】例如活动、营销、IT 设备、软件服务、物流、办公、生产物资等。
- 【偏好结构来源】用户当前说明、历史会话记录、上传模板、公司制度、检索证据或 domain pack 默认结构。
- 【建议采用的结构】说明本次应该按采购任务、规格参数、服务阶段、交付物、评分维度或供应商画像来组织。
- 【不适用结构】说明不建议套用的模板或字段，避免文档看似完整但不贴合真实采购。

## 三、需求规格与约束

补充会影响需求质量、验收、合同和后续策略/文档模板的关键要求：

- 【技术要求】针对产品/服务优化到参数或能力要求。
- 【质量标准】质量等级、服务标准、品牌/材质/资质等口径。
- 【验收口径】验收材料、验收节点、验收人或验收方法。
- 【交付方式】到场执行、分批交付、安装部署、邮寄配送等。
- 【发票类型】专票/普票/税率等。
- 【付款条款】预付款、尾款、验收后付款等。
- 【关键条款】保密、违约、延期、售后、知识产权、合规等。

## 四、需求完整性评估

始终区分：

- 已确认信息
- 当前假设 / 候选信息
- 待补充信息
- 待验证信息

待补充信息按优先级输出，只保留会影响预算、供应商选择、交付、验收、合同风险或后续需求分析、策略/文档模板的高价值缺口。

## 来源与信息状态

把进入正文的重要信息标注来源和状态，方便后续结构化页面做节点对齐与治理：

| 信息 | 来源 | 状态 | 使用方式 |
| --- | --- | --- | --- |
| 待补充 / 候选 | 用户当前输入 / 会话记录 / 上传模板 / 历史项目 / 检索资料 / domain pack | 已确认 / 候选 / 待验证 / 参考 | 写入正文 / 放入候选 / 进入待验证 |

## 五、决策口径与下一步

给出可执行的决策口径，而不是泛泛建议：

- 是否建议进入需求分析；说明依据。
- 是否建议进入 RFI / RFP / RFQ / RFX；说明适用条件。
- 推荐先确认的 1-3 个关键问题。
- 下一步需要准备的资料、模板或内部确认动作。
